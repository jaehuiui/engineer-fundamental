# Internet Communication — NAT, Port, Public IP

인터넷의 실제 통신 구조. IP, Port, NAT, 서버 아키텍처까지.

## Public IP vs Private IP

IPv4 주소는 약 43억 개뿐인데, 전 세계 인터넷 연결 장치는 수백억 개다.
해결책: 내부 네트워크에서는 사설 IP를 쓰고, 외부와 통신할 때만 공인 IP를 공유한다.

```
사설 IP 대역 (RFC 1918):
  10.0.0.0    ~ 10.255.255.255    (10.0.0.0/8)      ← 대규모 네트워크
  172.16.0.0  ~ 172.31.255.255    (172.16.0.0/12)    ← 중규모
  192.168.0.0 ~ 192.168.255.255   (192.168.0.0/16)   ← 가정/소규모

공인 IP: ISP가 할당, 인터넷에서 유일
사설 IP: 내부 네트워크에서만 유효, 다른 네트워크와 중복 가능
```

```
집 A: 192.168.0.10 (내 폰)
집 B: 192.168.0.10 (다른 사람 폰)
→ 같은 사설 IP지만 다른 네트워크이므로 충돌 없음
```

## NAT — Network Address Translation

사설 IP ↔ 공인 IP를 변환해주는 기술. 공유기(라우터)가 이 역할을 한다.

### NAT의 "뒤"와 "앞"

```
인터넷 (공인 IP 세계)          NAT (공유기)          내부 네트워크 (사설 IP 세계)
━━━━━━━━━━━━━━━━━━━━     ┃    ━━━━━━━━━━━━━━━━━━━━━━
                         ┃
  google.com             ┃      192.168.0.10 (내 PC)
  github.com        ←──  ┃  ──→ 192.168.0.11 (내 폰)
  외부 서버들               ┃      192.168.0.12 (TV)
                         ┃
  "NAT 앞"                ┃      "NAT 뒤"
  = 외부, 공인 IP 쪽        ┃      = 내부, 사설 IP 쪽
```

"NAT 뒤에 있다" = 사설 IP를 쓰고 있어서 외부에서 직접 접근할 수 없다는 뜻.

**왜 외부에서 접근이 안 되는가?**
```
내부 → 외부: 가능
  192.168.0.10 → NAT 변환 → 203.0.113.1 → google.com
  (NAT 테이블에 기록됨, 응답이 돌아올 수 있음)

외부 → 내부: 불가능
  외부 서버 → 203.0.113.1:??? → NAT 테이블에 매칭 없음 → 폐기!
  (누가 요청한 건지 모르니까 어디로 보내야 할지 알 수 없음)
```

NAT는 내부에서 먼저 요청을 보내야 테이블에 기록이 생기고, 그 기록을 보고 응답을 돌려보낸다.
외부에서 먼저 연결을 시도하면 NAT 테이블에 매칭되는 항목이 없어서 패킷이 버려진다.

**기본 동작:**
```
[내 PC]              [공유기/라우터]              [인터넷]           [서버]
192.168.0.10:54321 → 공인IP 203.0.113.1:54321 → ─────────── → 142.250.196.110:443
                      (NAT 변환)
```

공유기가 하는 일:
1. 내부 장치의 사설 IP를 공인 IP로 바꿔서 외부로 전송
2. 응답이 돌아오면 다시 사설 IP로 변환해서 내부 장치에 전달

### NAT Table

공유기는 "어떤 내부 장치가 어떤 외부 서버와 통신 중인지" 기록하는 테이블을 유지한다.

```
NAT Table:
┌──────────────────────┬───────────────────────┬──────────────────────┐
│ Internal             │ External (NAT)        │ Destination          │
├──────────────────────┼───────────────────────┼──────────────────────┤
│ 192.168.0.10:54321   │ 203.0.113.1:54321     │ 142.250.196.110:443  │
│ 192.168.0.11:49152   │ 203.0.113.1:49152     │ 151.101.1.69:443     │
│ 192.168.0.10:54322   │ 203.0.113.1:54322     │ 104.16.132.229:80    │
└──────────────────────┴───────────────────────┴──────────────────────┘
```

같은 내부 장치(192.168.0.10)라도 포트가 다르면 별도 엔트리.
이게 PAT(Port Address Translation)이다.

### NAT의 종류

| 종류 | 설명 | 용도 |
|------|------|------|
| SNAT (Source NAT) | 출발지 IP 변환. 내부 → 외부 통신 | 가정/회사 인터넷 접속 |
| DNAT (Destination NAT) | 목적지 IP 변환. 외부 → 내부 통신 | 포트 포워딩, 서버 공개 |
| PAT (Port Address Translation) | 하나의 공인 IP를 포트로 구분해서 여러 장치가 공유 | 가정용 공유기 (가장 흔함) |

### NAT의 한계

- 외부에서 내부로 먼저 연결할 수 없음 (서버 운영 어려움)
- P2P 통신이 어려움 (양쪽 다 NAT 뒤에 있으면)
- 일부 프로토콜이 IP를 payload에 포함하면 깨짐 (FTP active mode 등)

## Port — 프로세스 식별자

IP가 "어떤 컴퓨터"를 식별한다면, Port는 "그 컴퓨터의 어떤 프로그램"을 식별한다.

```
0 ~ 65535 (16비트)

Well-known Ports (0~1023):
  80   → HTTP
  443  → HTTPS
  22   → SSH
  53   → DNS
  25   → SMTP
  3306 → MySQL
  5432 → PostgreSQL
  6379 → Redis

Registered Ports (1024~49151):
  3000 → 개발 서버 (관례)
  8080 → 대체 HTTP

Dynamic/Ephemeral Ports (49152~65535):
  → 클라이언트가 자동으로 할당받는 임시 포트
```

**왜 클라이언트도 포트가 필요한가?**
```
브라우저 탭 1: 192.168.0.10:54321 → google.com:443
브라우저 탭 2: 192.168.0.10:54322 → google.com:443
브라우저 탭 3: 192.168.0.10:54323 → youtube.com:443

→ 같은 서버에 여러 연결이 있어도 클라이언트 포트로 구분
```

**Socket = IP + Port + Protocol**
```
TCP 소켓: (192.168.0.10, 54321, TCP) ↔ (142.250.196.110, 443, TCP)
이 조합이 하나의 연결을 고유하게 식별
```

## IPv4 vs IPv6

### IPv4

```
32비트, 약 43억 개 주소
표기: 10진수 4옥텟 (192.168.1.1)
헤더: 20~60 바이트 (가변)
```

**서브넷 마스크와 CIDR:**
```
192.168.1.0/24
  → 네트워크 부분: 192.168.1 (앞 24비트)
  → 호스트 부분: .0 ~ .255 (뒤 8비트, 256개 주소)
  → 사용 가능: .1 ~ .254 (0은 네트워크, 255는 브로드캐스트)

10.0.0.0/8    → 약 1677만 개 호스트
172.16.0.0/12 → 약 100만 개 호스트
192.168.0.0/16 → 약 6만 5천 개 호스트
```

**IPv4 고갈 문제:**
- 2011년 IANA의 IPv4 주소 풀 소진
- NAT, CIDR로 수명 연장했지만 근본적 해결은 아님
- IPv6로의 전환이 필요한 이유

### IPv6

```
128비트, 약 3.4 × 10^38 개 주소 (사실상 무한)
표기: 16진수 8그룹 (2001:0db8:85a3:0000:0000:8a2e:0370:7334)
축약: 연속된 0 그룹은 :: 으로 생략 (2001:db8:85a3::8a2e:370:7334)
헤더: 40 바이트 (고정)
```

**IPv4 vs IPv6 비교:**

| | IPv4 | IPv6 |
|---|---|---|
| 주소 길이 | 32비트 | 128비트 |
| 주소 수 | ~43억 | ~3.4 × 10^38 |
| 표기 | 10진수 (192.168.1.1) | 16진수 (2001:db8::1) |
| 헤더 | 가변 (20~60B) | 고정 (40B) |
| NAT | 필수 | 불필요 (모든 장치에 공인 주소) |
| IPSec | 선택 | 기본 내장 |
| 브로드캐스트 | 있음 | 없음 (멀티캐스트로 대체) |
| 자동 설정 | DHCP 필요 | SLAAC (자동 주소 설정) |
| 체크섬 | 헤더에 포함 | 제거 (상위 계층에 위임) |

**IPv6가 NAT를 불필요하게 만드는 이유:**
```
IPv4: 공인 IP 1개 → NAT → 사설 IP 여러 개
IPv6: 모든 장치가 고유한 공인 주소를 가짐
  → NAT 없이 end-to-end 통신 가능
  → P2P, IoT에 유리
```

**Dual Stack — 현실적 전환 방식:**
```
대부분의 시스템은 IPv4와 IPv6를 동시에 운영:
  서버: IPv4 (93.184.216.34) + IPv6 (2606:2800:220:1:248:1893:25c8:1946)
  클라이언트: IPv6 우선 시도 → 실패 시 IPv4 fallback (Happy Eyeballs 알고리즘)
```

## 서버의 통신 구조

### 단일 서버

```
클라이언트 A ──┐
클라이언트 B ──┼──→ 서버 (203.0.113.50:443)
클라이언트 C ──┘

서버는 하나의 포트(443)에서 수천 개의 동시 연결을 처리.
각 연결은 (클라이언트IP, 클라이언트Port, 서버IP, 서버Port) 4-tuple로 구분.
```

### 포트 포워딩 (DNAT)

서버가 NAT 뒤에 있을 때, 외부에서 접근할 수 있게 하는 방법.
NAT 테이블에 "영구적인 규칙"을 수동으로 추가하는 것이다.

```
포트 포워딩 없을 때:
  외부 → 203.0.113.1:80 → NAT: "누구한테 보내지?" → 폐기

포트 포워딩 설정 후:
  외부 → 203.0.113.1:80 → NAT: "80번은 192.168.0.100:3000으로!" → 내부 서버 도달
```

```
NAT 테이블 (자동 생성 — 임시):
  192.168.0.10:54321 ↔ 203.0.113.1:54321 → google.com:443  (통신 끝나면 삭제)

포트 포워딩 규칙 (수동 설정 — 영구):
  외부:80  → 192.168.0.100:3000  (항상 유효)
  외부:22  → 192.168.0.101:22    (항상 유효)
```

**공유기 관리 페이지에서 설정:**
```
┌──────────┬──────────┬─────────────────┬─────────────┐
│ 외부 포트  │ 프로토콜    │ 내부 IP          │ 내부 포트     │
├──────────┼──────────┼─────────────────┼─────────────┤
│ 80       │ TCP      │ 192.168.0.100   │ 3000        │
│ 443      │ TCP      │ 192.168.0.100   │ 3000        │
│ 22       │ TCP      │ 192.168.0.101   │ 22          │
│ 25565    │ TCP      │ 192.168.0.102   │ 25565       │
└──────────┴──────────┴─────────────────┴─────────────┘

→ 외부에서 203.0.113.1:80 접속 → 내부 192.168.0.100:3000 (웹 서버)
→ 외부에서 203.0.113.1:22 접속 → 내부 192.168.0.101:22 (SSH)
→ 외부에서 203.0.113.1:25565 접속 → 내부 192.168.0.102:25565 (마인크래프트)
```

### 포트 포워딩의 한계와 대안

```
한계:
  - 공인 IP 1개에 같은 포트는 하나의 내부 서버만 연결 가능
    (80번 포트를 두 서버에 동시에 포워딩 불가)
  - 공인 IP가 바뀌면 (유동 IP) 외부에서 접근 불가
  - 보안 위험: 내부 서버가 인터넷에 직접 노출

대안:
  - 리버스 프록시: nginx가 80/443을 받아서 도메인별로 다른 내부 서버로 분배
  - VPN: 외부에서 내부 네트워크에 안전하게 접속
  - 클라우드 터널: Cloudflare Tunnel, ngrok 등 (포트 포워딩 없이 외부 노출)
  - DDNS: 유동 IP 문제 해결 (IP 바뀔 때마다 DNS 자동 업데이트)
```

### 리버스 프록시 vs 포트 포워딩

```
포트 포워딩:
  외부:80 → 192.168.0.100:3000 (1:1 매핑, 하나만 가능)

리버스 프록시 (nginx):
  외부:80 → nginx (192.168.0.100:80)
              ├─ api.example.com  → 192.168.0.101:3000
              ├─ web.example.com  → 192.168.0.102:8080
              └─ blog.example.com → 192.168.0.103:4000
  → 하나의 포트로 여러 서비스를 도메인 기반으로 분배
```

### Reverse Proxy vs Load Balancer

구조적으로 매우 비슷하다. 둘 다 "앞에서 요청을 받아서 뒤에 있는 서버로 전달"하는 패턴.
차이는 목적과 기능 범위에 있다.

**리버스 프록시** — 클라이언트 대신 백엔드에 요청을 전달하는 대리인:
```
클라이언트 → nginx (리버스 프록시)
               ├─ api.example.com  → 백엔드 A (1대)
               ├─ web.example.com  → 백엔드 B (1대)
               └─ static 파일      → 직접 응답 (백엔드 안 거침)
```
역할: 도메인/경로 기반 라우팅, SSL 종료, 정적 파일 서빙, 캐싱, 백엔드 은닉

**로드 밸런서** — 같은 역할을 하는 여러 서버에 트래픽을 분산:
```
클라이언트 → 로드 밸런서
               ├─ 서버 1 (같은 앱)
               ├─ 서버 2 (같은 앱)
               └─ 서버 3 (같은 앱)
```
역할: 트래픽 분산, 헬스 체크 (죽은 서버 자동 제외), 세션 유지, 수평 확장

| | 리버스 프록시 | 로드 밸런서 |
|---|---|---|
| 목적 | 요청을 적절한 서버로 라우팅 | 트래픽을 여러 서버에 분산 |
| 백엔드 수 | 서비스당 1대도 가능 | 같은 서비스 여러 대 |
| 라우팅 기준 | 도메인, 경로, 헤더 등 | 알고리즘 (round-robin, least-conn 등) |
| 부가 기능 | SSL, 캐싱, 압축, 정적 파일 | 헬스 체크, failover, sticky session |

**실제로는 둘 다 한다 — nginx 예시:**
```nginx
upstream api_servers {
    least_conn;                    # 연결 수가 가장 적은 서버로
    server 10.0.1.10:3000;         # 서버 1
    server 10.0.1.11:3000;         # 서버 2
    server 10.0.1.12:3000 backup;  # 서버 3 (백업)
}

server {
    listen 443 ssl;

    # 리버스 프록시 + 로드 밸런싱 동시에
    location /api/ {
        proxy_pass http://api_servers;
    }

    # 정적 파일은 직접 서빙 (리버스 프록시 역할)
    location /static/ {
        root /var/www;
    }
}
```

**로드 밸런싱 알고리즘:**
```
Round Robin (기본):
  요청1 → 서버1, 요청2 → 서버2, 요청3 → 서버3, 요청4 → 서버1 ...

Least Connections:
  현재 연결 수가 가장 적은 서버로 → 처리 시간이 다른 경우 유리

IP Hash:
  클라이언트 IP 기반으로 항상 같은 서버로 → 세션 유지에 유리

Weighted:
  서버 성능에 따라 가중치 부여
  server 10.0.1.10 weight=3;  # 3배 더 많이 받음
  server 10.0.1.11 weight=1;
```

**L4 vs L7 로드 밸런싱:**
```
L4 (Transport): IP/Port만 보고 분산. 빠르지만 HTTP 내용 모름 (AWS NLB, HAProxy)
L7 (Application): HTTP 헤더, 경로, 쿠키까지 보고 분산. 느리지만 정밀 (nginx, ALB, Envoy)
```

### 실제 서비스의 통신 구조

```
사용자 브라우저
    │
    │ DNS 조회: example.com → CDN IP
    ↓
[CDN / Edge Server]  (CloudFront, Cloudflare 등)
    │
    │ 캐시 miss 시
    ↓
[Load Balancer]  (공인 IP, 하나의 진입점)
    │
    ├──→ [서버 1] 10.0.1.10  (사설 IP)
    ├──→ [서버 2] 10.0.1.11
    └──→ [서버 3] 10.0.1.12
              │
              ↓
         [DB / Cache]  10.0.2.x (사설 IP, 외부 접근 불가)
```

- CDN: 정적 자산 캐시, 사용자와 가까운 엣지에서 응답
- Load Balancer: 공인 IP를 가지고, 트래픽을 내부 서버들에 분배
- 서버들: 사설 IP, 외부에서 직접 접근 불가
- DB: 더 깊은 사설 네트워크, 서버에서만 접근 가능

## 전체 통신 흐름 예시

브라우저에서 `https://example.com`을 입력했을 때:

```
1. DNS 조회
   브라우저 → DNS 서버 (UDP 53)
   example.com → 93.184.216.34

2. TCP 연결 (3-Way Handshake)
   내 PC (192.168.0.10:54321)
     → NAT 변환 → 공인IP (203.0.113.1:54321)
       → 인터넷 라우팅
         → 서버 (93.184.216.34:443)
   SYN → SYN-ACK → ACK

3. TLS Handshake (HTTPS)
   인증서 검증, 암호화 키 교환
   → 이후 모든 데이터 암호화

4. HTTP 요청
   GET / HTTP/1.1
   Host: example.com

5. HTTP 응답
   서버 (93.184.216.34:443)
     → 인터넷 라우팅
       → 공유기 (203.0.113.1:54321)
         → NAT 역변환 → 내 PC (192.168.0.10:54321)
   HTTP/1.1 200 OK
   <html>...</html>

6. 렌더링
   브라우저가 HTML 파싱 → 추가 리소스 요청 (CSS, JS, 이미지)
   → 각각 별도의 TCP 연결 또는 HTTP/2 스트림
```

## NAT Traversal — NAT 뒤의 장치끼리 통신하기

양쪽 다 NAT 뒤에 있으면 서로 직접 연결할 수 없다. WebRTC 등 P2P 통신에서 이 문제를 해결해야 한다.

**STUN (Session Traversal Utilities for NAT):**
```
[A: NAT 뒤] → STUN 서버에 요청 → "너의 공인 IP:Port는 203.0.113.1:54321이야"
[B: NAT 뒤] → STUN 서버에 요청 → "너의 공인 IP:Port는 198.51.100.1:49152야"
→ 서로의 공인 주소를 알게 되면 직접 연결 시도 (hole punching)
```

**TURN (Traversal Using Relays around NAT):**
```
STUN으로도 안 되면 (Symmetric NAT 등), 중계 서버를 통해 통신:
[A] → TURN 서버 → [B]
```

**ICE (Interactive Connectivity Establishment):**
- STUN과 TURN을 조합해서 최적의 연결 경로를 찾는 프레임워크
- WebRTC가 내부적으로 ICE를 사용

## NAT 뒤에서 개인 서버 운영하기

NAT 때문에 기본적으로 외부에서 접근이 안 되지만, 여러 방법으로 우회할 수 있다.

### 1. 포트 포워딩 + DDNS (가장 전통적)

```
문제: ISP가 유동 공인 IP를 줌 → IP가 바뀌면 접속 불가
해결: DDNS (Dynamic DNS)

DDNS 동작:
  1. DDNS 서비스에 가입 (no-ip.com, duckdns.org 등)
  2. myserver.duckdns.org 같은 도메인을 받음
  3. 집 PC/공유기에서 주기적으로 현재 공인 IP를 DDNS 서버에 알림
  4. 공인 IP가 바뀌어도 도메인은 항상 최신 IP를 가리킴

myserver.duckdns.org → DNS 조회 → 현재 공인 IP → 포트 포워딩 → 내부 서버
```

한계:
- ISP가 포트를 막는 경우가 있음 (80, 443 등 well-known 포트 차단)
- 이중 NAT 문제 (ISP 공유기 + 내 공유기 → 포트 포워딩 2번 필요)
- 보안 직접 관리 필요

### 2. 클라우드 터널 (포트 포워딩 없이)

내부에서 외부로 먼저 연결을 열고, 그 터널을 통해 외부 트래픽을 받는 방식.
NAT의 "내부 → 외부는 가능" 특성을 역이용한다.

```
기존 포트 포워딩:
  외부 → NAT → 내부 (NAT가 막음, 포트 포워딩 필요)

터널 방식:
  내부 서버 → 터널 서비스에 연결 (내부→외부이므로 NAT 통과!)
  외부 사용자 → 터널 서비스 → 이미 열린 터널 → 내부 서버

  [내 서버] ──연결──→ [Cloudflare/ngrok] ←──접속── [외부 사용자]
            (NAT 통과)    (공인 IP 보유)
```

```bash
# ngrok
ngrok http 3000
# → https://abc123.ngrok.io 로 외부 접속 가능

# Cloudflare Tunnel
cloudflared tunnel --url http://localhost:3000
# → mysite.example.com 으로 외부 접속 가능 (자체 도메인 연결, 무료)
```

### 3. VPN / Tailscale

서버를 공개하지 않고, 특정 기기만 접근하게 하는 방식.

```
Tailscale (WireGuard 기반):
  내 폰, 노트북, 집 서버 모두 Tailscale 설치
  → 각 기기에 가상 IP 부여 (100.x.x.x)
  → NAT 뒤에 있어도 기기끼리 직접 통신 가능 (NAT traversal 자동)

  [내 폰: 100.64.0.1] ←──→ [집 서버: 100.64.0.2]
  카페 Wi-Fi 뒤에 있어도     집 NAT 뒤에 있어도
  서로 직접 연결됨
```

### 4. 클라우드 VPS + 리버스 프록시

집 서버를 직접 노출하지 않고, 저렴한 클라우드 서버를 중계점으로 사용.

```
[외부 사용자] → [클라우드 VPS: 공인 IP] → SSH 터널/VPN → [집 서버]

VPS (월 $3~5):
  - 공인 IP 고정
  - nginx 리버스 프록시
  - SSL 인증서 관리
  - 집 서버와 SSH 터널 또는 WireGuard로 연결
```

### 방식 비교

| 방식 | 난이도 | 비용 | 포트 포워딩 필요 | 고정 도메인 |
|------|--------|------|-----------------|------------|
| 포트 포워딩 + DDNS | 중 | 무료 | O | DDNS 도메인 |
| ngrok | 하 | 무료/유료 | X | 유료만 고정 |
| Cloudflare Tunnel | 중 | 무료 | X | 자체 도메인 |
| Tailscale | 하 | 무료 | X | 내부 전용 |
| VPS + 터널 | 상 | 월 $3~5 | X | 자체 도메인 |
